# Session 2026-07-04 — T-011 Watchlist + Realtime Alerts

## What was done
- Models: Watchlist (4 list types, is_active, unique name), WatchlistEntry
  (person PROTECT, severity, active, unique watchlist+person), WatchlistAlert
  (entry+trigger_job unique, score, message, ack fields, indexed
  (acknowledged, created_at)). All audited.
- Hook: `apps.matching.signals.match_job_completed` (new django Signal, sent
  by execute_job after DONE); watchlist receiver (AppConfig.ready) →
  `create_alerts_for_job`: one alert per (active entry, active list, job)
  with best candidate score per person; push to Channels group "alerts".
  Latent candidates (person null) skipped naturally. Alerts fire on
  candidates (machine matches), not human decisions — realtime is the point
  (ADR-020). Dedup jobs alert too: watchlisted person re-enrolling triggers.
- Channels: `AlertConsumer` (AsyncJsonWebsocketConsumer; role gate
  admin/investigator/supervisor, close 4403, group add/discard, alert.created
  → send_json). `accounts.ws_auth.JWTAuthMiddleware` (?token=<access>,
  role force-loaded so the consumer never touches the DB). asgi.py:
  AllowedHostsOriginValidator(JWTAuthMiddleware(URLRouter(...))).
  config/routing.py registers ws/alerts/.
- Endpoints: watchlists CRUD (DELETE → 405; deactivate via PATCH), nested
  entries GET/POST + PATCH/DELETE(entry deactivate) at
  /watchlists/{id}/entries/{entry_id}/, alerts read-only + POST ack
  (idempotent — second ack keeps original timestamp). WatchlistPermission:
  inv/sup/admin all ops; operator/auditor 403.
- Gotcha fixed: viewset `http_method_names` without "delete" blocks DELETE
  on @action routes too — allow at dispatch, 405 in destroy().
